## What this is

Degen Dogs Mission 3 Analytics tracks the Mission 3 auction feed, current dog, recent winners, bidder activity, WOOF holders, Farcaster identity resolution, and Season 5 SUP reward estimates.

It was built because the Mission 3 workflow needed a custom, project-specific dashboard instead of relying on a generic Dune page. The repo keeps the analytics logic close to the generated artifacts so the community can inspect the query layer and download the data behind the site.

The public site is a cached GitHub Pages dashboard backed by generated CSV/JSON files. It does not expose a public SQL editor; all queries are approved and executed by the local data runner before publishing.

The architecture is intentionally simple: local runner -> SQLite query layer -> CSV/JSON/static site -> GitHub Pages.

```text
Base RPC + contract calls
        ↓
Mac mini / local runner
        ↓
Python decoder + in-memory SQLite
        ↓
sql/mission3_dashboard.sql
        ↓
generated CSV/JSON + index.html
        ↓
GitHub Pages dashboard
```

## How to use this dashboard

- Check the top card for the current dog, current high bid, and countdown.
- Use the auction feed to see the current auction and recent winners.
- Click Dog images in the auction feed to open their OpenSea item pages.
- Click current-Dog traits to filter the Degen Dogs OpenSea collection.
- Use traits and rarity to understand the dog's profile.
- Use the advanced tables or downloads only if you want raw data.
- The dashboard updates when the private runner publishes a new snapshot.

## How the data pipeline works

1. Fetch Base auction-house logs and WOOF transfer logs.
2. Fetch current auction and contract state through RPC calls.
3. Resolve Dog metadata, rarity, images, and Farcaster identity data where available.
4. Load decoded data into local SQLite tables.
5. Execute [`sql/mission3_dashboard.sql`](sql/mission3_dashboard.sql).
6. Export approved result tables to `generated/*.csv` and `generated/*.json`.
7. Rebuild the static dashboard and publish through GitHub Pages.

## Local development

```bash
npm ci
npm run data
npm run dev
npm run build
```

- `npm run data` refreshes Base data, executes the SQL layer, and regenerates `generated/`, `public/generated/`, `index.html`, and `README.md`.
- `npm run dev` previews the Vite site locally.
- `npm run build` creates the production build in `dist/`.

## Private runner / refresh model

The intended production flow is a private Mac mini or local machine running the refresh pipeline on a schedule. The runner fetches data, executes the SQL, regenerates files, commits/pushes changes, and GitHub Pages serves the built static site.

If the local runner has been configured, these helpers are available:

```bash
npm run refresh:publish
npm run refresh:install
```

- `npm run refresh:publish` runs the refresh/publish flow from the local machine.
- `npm run refresh:install` installs or refreshes the local scheduled runner configuration.

These commands assume local secrets, git auth, and runner environment variables are already configured. The Mac mini/private runner is not the public host.

## Configuration

Supported environment variables used by `scripts/build_dashboard.py`:

| Variable | Purpose |
| --- | --- |
| `BASE_RPC_URL` | Single Base RPC endpoint for contract calls; also overrides log RPC lists when set. |
| `BASE_RPC_URLS` | Comma-separated fallback Base RPC endpoints for contract calls. |
| `BASE_LOG_RPC_URLS` | Comma-separated Base RPC endpoints used for `eth_getLogs` history scans. |
| `BASE_FROM_BLOCK` | First Base block scanned for Mission 3 logs; defaults to the known Mission 3 start range. |
| `BASE_LOG_CHUNK` | Maximum block range per log request, capped at 10,000 for public Base RPC compatibility. |
| `BASE_LOG_WORKERS` | Concurrent log-fetch workers, capped by the builder to avoid public RPC overload. |
| `BASE_RPC_BATCH_LIMIT` | Maximum JSON-RPC batch size for balance/metadata calls, capped at 10. |
| `DOG_METADATA_WORKERS` | Concurrent Dog metadata fetch workers, capped by the builder. |
| `NEYNAR_API_KEY` | Optional Neynar API key for Farcaster identity resolution. |
| `WOOF_USD_PRICE` | Optional manual WOOF/USD override; otherwise fetched from Dexscreener Base pools. |
| `SUP_USD_PRICE` | Optional manual SUP/USD override; otherwise fetched from Dexscreener Base pools. |

## Trust and caveats

- The dashboard is a cached snapshot, not a real-time database.
- Visitor-run SQL is intentionally not exposed on the public site.
- Current-auction state comes from on-chain contract calls.
- Historical auction rows are reconstructed from Base event logs.
- Farcaster identity resolution may be incomplete.
- Season 5 SUP reward tables are estimates unless/until confirmed against official reward logic.
- Archived Dune SQL bundles are historical/provenance material and may include reconstructed or partial SQL.

## For developers: where to change things

- `sql/mission3_dashboard.sql`: analytics query logic and output table definitions.
- `scripts/build_dashboard.py`: RPC fetching, event decoding, exports, README generation, and site generation.
- `README.template.md`: stable human-written README structure.
- `generated/`: generated output artifacts; do not hand-edit these long term.

## Adding a new metric

1. Add or modify the required decoded input in `scripts/build_dashboard.py` if needed.
2. Add a new `CREATE TABLE` statement in `sql/mission3_dashboard.sql`.
3. Add the table to `OUTPUT_TABLES` in the builder if it should be exported.
4. Run `npm run data` and verify the generated CSV/JSON output.
5. Run `npm run build` and preview the site.
